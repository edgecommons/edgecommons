## The shape (the ordering is the archetype)

`Destination` is the interface you implement: `kind()`, `deliver(item)` (idempotent, to a stable
key), `verify(item, delivered)` (checks what actually landed, before confirming). Every failure is
classified transient or permanent (`DeliverException`); retry is exponential backoff with full
jitter against a **time budget** (`giveUpAfterMs`), not an attempt count. Every transition is
reported on the UNS `evt` class, and `delivery-exhausted` is critical — it means data did not arrive.
See `Destination.java`, `LocalDestination.java`, and `docs/explanation.md`.

## Config

Sink config lives under `component.global` / `component.instances[]`, validated by
`config.schema.json` and enforced strictly at runtime by `SinkConfig.parse` (an unknown key is a
startup error). The sibling sections (`hierarchy`, `identity`, `messaging`, `logging`, `heartbeat`,
`metricEmission`, `credentials`) are the canonical EdgeCommons config schema and are deliberately not
redeclared here.

## Org conventions this scaffold inherits

- **UNS grammar:** `ecv1/{device}/{component}/{instance}/{class}[/channel]`. The event ladder rides
  `evt/{severity}/{type}`, minted from the body's own severity + type via `getEvents()`.
- A destination must be idempotent to a stable key and must be verified before the source is
  released — these two properties are non-negotiable for any backend you add.
