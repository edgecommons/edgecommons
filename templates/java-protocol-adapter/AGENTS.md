## What this is

`<<COMPONENTNAME>>` (Greengrass component name `<<COMPONENTFULLNAME>>`) is a **southbound protocol
adapter** generated from the EdgeCommons `java-protocol-adapter` template. An adapter connects to
field devices over some protocol, publishes their signals onto the message bus as
`SouthboundSignalUpdate`, and serves an on-demand read/write/browse command surface — so a consumer
can chart a signal without knowing what protocol produced it. The scaffold ships with an in-process
simulated backend (`Device.SimBackend`) so it runs and is testable with no hardware; a real adapter
replaces that backend and keeps everything above the seam unchanged.

## The shape (the seam you implement)

`Device.java` defines `DeviceSession` (one live connection) and `DeviceBackend` (opens sessions). One
worker per configured `component.instances[]` entry owns that device's session, connects with
jittered exponential backoff, polls on `pollIntervalMs`, and reconnects on a dropped link. Everything
above the seam — `<<COMPONENTNAME>>.java` (wiring, the worker, connectivity), `Commands.java` (the
`sb/*` command surface + edge-console panels), `Metrics.java` (`southbound_health` + the operational
families) — is written against the interface and never learns your protocol. Replace `SimBackend`;
do not change the files above the seam to accommodate a specific protocol — extend the seam instead.

A backend knows protocols. It does not know EdgeCommons topics, envelopes, or metrics — `Device.java`
imports nothing from `com.mbreissi.edgecommons` on purpose. If a `DeviceSession` implementation
reaches for the UNS or a facade, the seam has leaked.

## Validation expectations

- `mvn test` must pass with no live infrastructure — the sim backend is what the unit tests exercise.
- `src/test/java/.../LiveSimIT.java` is a **live** integration test, gated on the `EC_LIVE_SIM`
  environment variable; it must self-skip (green, not run) in a normal `mvn test`.
- This component inherits the org's 90% line-coverage gate (JaCoCo `check`, wired into `pom.xml`,
  enforced by `mvn verify` in CI). Don't lower the gate or exclude testable code to pass it — add
  tests. The gate's `<excludes>` scope out only the thin live-runtime seam — the component's `main()`
  bootstrap and the `DeviceWorker` connect-with-backoff supervisor + poll loop, which need a live
  device session and are validated by `LiveSimIT` on real infrastructure — so every class with
  unit-testable logic (the `Wiring` helpers, the command surface, the metric emitter, the device
  seam, the config model) stays in the gate. Keep it that way: extract logic out of the run loop
  rather than excluding it.
- Public behavior changes to the command surface or metric families should stay consistent with the
  southbound contract described in `docs/reference/messaging-interface.md` and `docs/reference/metrics.md`.

## Org conventions this scaffold inherits

- **UNS grammar:** `ecv1/{device}/{component}/{instance}/{class}[/channel]`. Reserved classes
  (`state`, `metric`, `cfg`, `log`) are library-owned; publish through the library's facades
  (`data()`, `events()`), never by hand-building a topic or envelope.
- **Southbound contract:** a data point is a **signal**, not a tag. `SouthboundSignalUpdate` quality
  is normalized to `GOOD | BAD | UNCERTAIN`, with the native code preserved in `qualityRaw`.
- **Writes are allow-listed** by stable `signal.id`, checked before any device I/O. An empty
  `writes.allow` means read-only, which is the correct default for anything touching a control
  system.
- **Builders are the construction path** — `MetricBuilder`, `MessageBuilder`, the `data()`/`events()`
  facades — not raw constructors, for new code you add.
