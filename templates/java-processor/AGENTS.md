## The shape (one route per instance)

Each `component.instances[]` entry is one route: topic filters, a `Pipeline` of `Processor` stages, a
publish topic, and a target (`local`/`northbound`). A stage returns zero or more messages; a stateful
stage also implements `onTick`. Two guards are load-bearing and must not be removed: the **self-echo
guard** (drop anything carrying our own identity) and the **identity restamp** (every published
message is rebuilt with `.withConfig(...)` before it goes out). See `Stages.java`, `RouteConfig.java`,
and `docs/explanation.md`.

## Config

Route config lives under `component.global` / `component.instances[]`, validated by
`config.schema.json` and enforced strictly at runtime by `RouteConfig.parse` (an unknown key is a
startup error). The sibling sections (`hierarchy`, `identity`, `messaging`, `logging`, `heartbeat`,
`metricEmission`, `credentials`) are the canonical EdgeCommons config schema and are deliberately not
redeclared here.

## Org conventions this scaffold inherits

- **UNS grammar:** `ecv1/{device}/{component}/{instance}/{class}[/channel]`. A route's
  `publishTopic` must name an application class; reserved classes are library-owned and rejected on
  direct publish.
- A processor uses raw `getMessaging()`, not the `data()` facade — it is payload-agnostic and
  republishes what it was handed, on a topic its config names. See `docs/explanation.md`.
