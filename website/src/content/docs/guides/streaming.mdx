Telemetry streaming is the SDK's high-rate, one-way egress path. Your code appends records to an
embedded, persistent (or in-memory) buffer; a background export engine batches them, retries on
failure, and checkpoints on acknowledgement to **Kinesis** or **Kafka**. The buffer keeps accepting
records while the network or sink is down, so a cloud disconnect never blocks your hot loop or loses
durably-buffered data.

All four languages bind the **same shared Rust core** (`edgestreamlog`): Rust links it directly, while
Java (FFM / Panama), Python (PyO3), and TypeScript (napi-rs) reach it through native bindings.

<Aside type="note" title="Opt-in subsystem">
Streaming is inactive unless your component config has a `streaming` section. The accessor returns
`null` / `None` / `undefined` (Java / Python / TypeScript) when there is no `streaming` config, and
in Rust the whole module is behind the off-by-default `streaming` cargo feature.
</Aside>

## When to use it

Reach for streaming when you have **high-rate, fire-and-forget telemetry** (sensor readings, metrics,
events) that must reach Kinesis or Kafka with **at-least-once** delivery and must **survive
intermittent connectivity**. It is deliberately one-way: there is no request/reply and no
server-to-device path. For command-and-control or low-rate messaging, use the
[Messaging guide](/guides/messaging/) instead.

## Enable streaming


1. **Add a `streaming` section to your component config.** Each entry under `streaming.streams[]`
   requires a `name`, a `sink`, **and a `buffer`**; `batch` and `delivery` are optional and default
   to sensible values.

   <Aside type="caution" title="buffer is required by the core (but the schema doesn't enforce it)">
   The shared `edgestreamlog` core deserializes `buffer` as a **required** field. A stream with no
   `buffer` **fails to open** тАФ the core reports a missing `buffer` field and the whole streaming
   subsystem fails to start. The JSON schema, however, only requires `["name", "sink"]`, so a
   buffer-less config **passes schema validation and then fails at runtime**. This is a known
   schema тЖФ core gap: always supply `buffer`. (`batch` and `delivery` genuinely are optional тАФ they
   have defaults. Inside `buffer`, the only field the core itself requires is `path`, and only for a
   `disk` buffer; the rest of the buffer fields default.)
   </Aside>

   ```json
   {
     "component": { "name": "com.example.Telemetry" },
     "streaming": {
       "streams": [
         {
           "name": "telemetry",
           "sink": { "type": "kinesis", "streamName": "edge-{ThingName}" },
           "buffer": {
             "type": "disk",
             "path": "/var/lib/edgecommons/streams/telemetry",
             "maxDiskBytes": 1073741824,
             "onFull": "block",
             "fsync": "perBatch"
           }
         }
       ]
     }
   }
   ```

   Template variables (`{ThingName}`, `{ComponentName}`) in `buffer.path`, the Kinesis `streamName`,
   and the Kafka `bootstrapServers` / `topic` are resolved before the stream opens. A
   `{"$secret": ...}` reference anywhere in the `streaming` section is resolved from the
   [credentials vault](/guides/credentials/) first.

   In hierarchical-config deployments, `streaming.streams` may live in an inherited hierarchy layer. Those entries
   are still configuration only: every component opens its own `StreamService`, handles, buffers,
   drains, metrics, and shutdown path. Disk buffer paths must resolve to component-unique locations;
   include `{ComponentName}` or another per-component segment when streams are inherited.

2. **Make the native core available** for your language (see the table below).

3. **Get the stream service from the runtime** and append records (next section).


### Native-core prerequisites

| Language | How the core is provided | Runtime requirement |
|----------|--------------------------|---------------------|
| Java | cdylib `edgestreamlog` loaded once per process (via `-Dedgestreamlog.library.path`, `java.library.path`, or a bundled classpath resource) | Run with `--enable-native-access=ALL-UNNAMED` (the sink/log bridges use upcall stubs) |
| Python | PyO3 wheel `edgestreamlog_native`, imported lazily | Install the wheel; failure surfaces on first use as `EdgeStreamError(-1, ...)` |
| TypeScript | napi addon `@edgecommons/streamlog-node`, `require`-d lazily on first `open` | Install the addon; a component that never streams never loads it |
| Rust | Linked directly тАФ **compile-time** via cargo features | Build with `streaming` plus a sink feature (below) |

For Rust, the `streaming` feature enables the module and accessor; **real export also needs a sink
feature** тАФ `streaming-kinesis` and/or `streaming-kafka`. Without a sink feature, a stream still
buffers durably but never exports (see [Buffer-only fallback](#buffer-only-fallback)).

```toml
# Cargo.toml
[dependencies]
edgecommons = { version = "*", features = ["streaming", "streaming-kinesis"] }
```

## Get a stream and append records

In a component you get the service from the runtime accessor, fetch a `StreamHandle` by name, then
`append` / `flush` and optionally read `stats`. The runtime opens every configured stream for you and
closes them on shutdown.

<Aside type="caution" title="The append signature differs in Rust">
Java, Python, and TypeScript take three positional arguments тАФ
`append(partitionKey, timestampMs, payload)`. **Rust takes a single `Record` value** built with
`StreamRecord::new(partitionKey, timestampMs, payload)`, and additionally offers a Rust-only
`append_batch(&[Record])`.
</Aside>

<Tabs syncKey="lang">
<TabItem label="Java">

```java
StreamService streams = gg.getStreams();   // null if there is no `streaming` config
if (streams != null) {
    StreamHandle h = streams.stream("telemetry");   // throws EdgeStreamException (ERR_UNKNOWN_STREAM) if absent
    h.append("pump-7", System.currentTimeMillis(),
             "reading-0".getBytes(StandardCharsets.UTF_8));   // partitionKey <= 65535 bytes
    h.flush();   // force the buffer durably to disk (does NOT wait for export)

    StreamStats s = streams.stats("telemetry");
    log.info("appended={} backlog={}", s.appendedTotal(), s.backlog());
}
```

### Closing streams

When you use the runtime accessor, the SDK owns the service lifecycle and closes streams on shutdown.
If you open a `StreamService` yourself (for standalone testing or a custom buffer), close it to flush
all streams and stop the export engines. `close()` is idempotent.

<Aside type="note" title="Handle lifecycle differs in Rust">
In Java, Python, and TypeScript a `StreamHandle` has its own `close()` (and Java/Python integrate with
try-with-resources / the context-manager protocol). **Rust's `StreamHandle` is `Clone`/Arc-backed and
has no `close()`** тАФ it is dropped via RAII. In Java a handle even outlives its service: `append` and
`flush` keep working after `service.close()`.
</Aside>

<Tabs syncKey="lang">
<TabItem label="Java">

```java
String cfg = """
  {"streams":[{"name":"telemetry","sink":{"type":"kinesis","streamName":"x"},
   "buffer":{"path":"/tmp/telemetry","maxDiskBytes":1073741824,"onFull":"block"}}]}""";
try (StreamService svc = StreamService.open(cfg);
     StreamHandle h = svc.stream("telemetry")) {
    h.append("pump-7", 1000L, "reading-0".getBytes(StandardCharsets.UTF_8));
    h.flush();
}   // try-with-resources closes both
```

</TabItem>

## Sinks

A stream's `sink` is tagged by `type`. Two sink types are declarable in component config: `kinesis`
and `kafka`. The same config is used by all four languages.

### Kinesis

```json
{
  "name": "telemetry",
  "sink": {
    "type": "kinesis",
    "streamName": "edge-{ThingName}",
    "region": "us-east-1",
    "endpointUrl": "http://localhost:4566"
  }
}
```

| Field | Required | Notes |
|-------|----------|-------|
| `streamName` | yes | Target Kinesis data stream; supports template variables. |
| `region` | no | AWS region; falls back to the ambient AWS region when omitted. |
| `endpointUrl` | no | Override the endpoint for LocalStack / floci / a VPC endpoint. |

### Kafka

```json
{
  "name": "telemetry",
  "sink": {
    "type": "kafka",
    "bootstrapServers": "broker-1:9092,broker-2:9092",
    "topic": "edge.telemetry",
    "properties": { "compression.type": "lz4", "acks": "all" }
  }
}
```

| Field | Required | Notes |
|-------|----------|-------|
| `bootstrapServers` | yes | `host:port[,host:port...]`; supports template variables. |
| `topic` | yes | Target Kafka topic; supports template variables. |
| `properties` | no | Open map of librdkafka producer properties passed through verbatim. |

### File

Rolling **Parquet/AVRO** files on local disk, for later bulk upload to a cloud data lake (S3/Glue/Athena, Azure ADLS/Synapse, GCS/BigQuery). Build with the `streaming-file-parquet` and/or `streaming-file-avro` feature.

```json
{
  "name": "archive",
  "sink": {
    "type": "file",
    "format": "parquet",
    "mode": "rows",
    "dir": "/data/{ThingName}/archive",
    "partitionBy": "dt={yyyy-MM-dd}/hr={HH}",
    "maxFileBytes": 134217728,
    "maxFiles": 64,
    "rollEverySecs": 300,
    "compression": "snappy"
  }
}
```

| Field | Required | Notes |
|-------|----------|-------|
| `format` | no | `parquet` (default) or `avro`. |
| `mode` | no | `rows` (default) = typed columns via the `rows` projection (default = the `SouthboundSignalUpdate` layout, envelope `tags` as one JSON column); `raw` = one opaque row per message. With the default projection a non-`SouthboundSignalUpdate` payload lands in a sibling `_unmapped` file. |
| `dir` | yes | Output directory root; template variables resolved before the stream opens. |
| `partitionBy` | no | Hive-style sub-path with UTC time tokens (`{yyyy-MM-dd}`, `{HH}`, тАж) resolved per file at roll time. |
| `maxFileBytes` | no | Roll past this size (default 128 MiB). Soft cap at row-group granularity; keep `batch.maxBytes` below it for tight files. |
| `maxFiles` | no | Ring cap of finalized files (`0` = unbounded). |
| `rollEverySecs` | no | Also roll on this interval (`0` = disabled). |
| `onFull` | no | `dropOldest` (default) or `stop` when `maxFiles` is reached. |
| `compression` | no | `none` / `snappy` (default) / `zstd` / `gzip`. |
| `rows` | no | `rows`-mode column projection тАФ `explode` + `columns[]` (`name`/`path`/`type`). Absent тЖТ the default `SouthboundSignalUpdate` projection. |

<Aside type="note" title="rows mode vs raw mode">
`rows` columns are **declared by config** (the `rows` projection), so you map any message shape to a typed, query-ready table тАФ ideal for Athena/BigQuery/Synapse. The default projection (no `rows` block) mirrors `SouthboundSignalUpdate`: one row per sample, the envelope `tags` as a single JSON column, the value in sparse typed columns (Avro uses a true union). `raw` mode is format-agnostic archival (opaque payload) for anything you don't want to project. Delivery is at-least-once: a clean shutdown finalizes the open file; on a hard crash Avro recovers to its last block while an unclosed Parquet file is discarded. The [telemetry-processor component](/components/telemetry-processor/) is the reference producer for this sink.
</Aside>

<Aside type="note" title="Callback sinks are programmatic only">
The core also supports a `callback` sink for bring-your-own delivery, but it is **not part of the
JSON schema** тАФ a config with `sink.type: "callback"` fails schema validation. Callback streams are
reachable only programmatically (Java `StreamService.registerSink`, Python
`StreamService.open_with_callback`, TypeScript `registerSinkCallback`, Rust
`DefaultStreamService::open_with`). A `callback` sink also accepts an optional `id`
(`{ "type": "callback", "id": "metrics-cw" }`) used to route to a specific host callback when several
callback streams coexist. Treat it as an internal extension point, not a config-declarable sink.
</Aside>

## The buffer

Each stream owns an embedded buffer that decouples your producer loop from the network. Configure it
under `buffer`.

### Disk vs memory

- **`disk`** (default) тАФ a durable, segmented on-disk log. `path` is **required**. Survives process
  restarts and powers store-and-forward across cloud disconnects.
- **`memory`** тАФ a non-durable in-RAM ring. `path` must be **omitted**. `maxDiskBytes` is reinterpreted
  as the in-RAM byte budget. Records are lost on restart тАФ use only when durability is not needed.

| Field | Default | Meaning |
|-------|---------|---------|
| `type` | `disk` | `disk` (durable) or `memory` (non-durable RAM ring). |
| `path` | тАФ | Required for `disk`; must be omitted for `memory`. |
| `segmentBytes` | `67108864` (64 MiB) | On-disk segment size. |
| `maxDiskBytes` | `1073741824` (1 GiB) | Total buffer budget (on disk, or in RAM for `memory`). Must be `>= segmentBytes`. |
| `maxAgeSecs` | _null_ | **No effect** (see below). |
| `onFull` | `dropOldest` | Behavior when the budget is reached (below). |
| `fsync` | `perBatch` | Durability flush policy: `always`, `perBatch`, or `interval`. |
| `fsyncIntervalMs` | `1000` | Flush cadence when `fsync: interval`. |
| `maxBufferedRecords` | `10000` | In-memory ingest-queue bound тАФ the actual backpressure point. |

<Aside type="caution" title="maxAgeSecs has no effect">
There is **no age-based retention**. The buffer enforces its budget purely by **bytes**: `dropOldest`
reclaims the oldest segment (disk) or record (memory) only when an append would push the buffer past
`maxDiskBytes`. `maxAgeSecs` is parsed but never read, so records are **never dropped because of
age** тАФ a buffer that stays under `maxDiskBytes` retains its data indefinitely no matter how old it
gets. `maxAgeSecs` is inert, so size your retention with `maxDiskBytes`.
</Aside>

### onFull semantics

The `onFull` policy decides what happens when the buffer budget is exhausted:

- **`dropOldest`** (default) тАФ never blocks the producer; drops the oldest records to stay within budget.
  Drops are metered (`droppedTotal`), never silent.
- **`block`** тАФ the producer **blocks** in `append` until the export engine reclaims space. Lossless.
- **`rejectNew`** тАФ `append` fails immediately with `ERR_FULL` / `BufferFull`.

<Aside type="caution" title="append is not unconditionally non-blocking">
Whether `append` blocks is **policy-dependent**. Under `onFull: block` it blocks until space frees;
under `fsync: always` it blocks until the record is fsync'd to disk. The "append never blocks your
loop" guarantee holds only for the default `dropOldest` policy.
</Aside>

### Batch and delivery tuning

These optional sections tune how the background engine batches and retries. The defaults are
production-ready; tune only when you have a reason.

```json
{
  "batch": { "maxRecords": 500, "maxBytes": 4194304, "maxLatencyMs": 1000, "compression": "none" },
  "delivery": { "maxRetries": -1, "backoffBaseMs": 50, "backoffMaxMs": 30000, "pollIntervalMs": 100 }
}
```

| Section.field | Default | Meaning |
|---------------|---------|---------|
| `batch.maxRecords` | `500` | Max records per export batch. |
| `batch.maxBytes` | `4194304` (4 MiB) | Max bytes per export batch. |
| `batch.maxLatencyMs` | `1000` | **No effect** (see below). |
| `batch.compression` | `none` | `none` or `zstd` (`zstd` is a no-op тАФ see below). |
| `delivery.maxRetries` | `-1` | Max attempts for a whole-batch `Failed` send before abandoning it (`-1` = retry forever). Bounds **only** whole-batch failures тАФ per-record partial failures retry indefinitely (see [Delivery semantics](#delivery-semantics)). |
| `delivery.backoffBaseMs` | `50` | Initial retry backoff. |
| `delivery.backoffMaxMs` | `30000` | Backoff ceiling. |
| `delivery.pollIntervalMs` | `100` | Export poll cadence тАФ also the effective partial-batch drain interval (see below). |

<Aside type="caution" title="batch.maxLatencyMs has no effect">
The export loop reads each batch with `read_batch(maxRecords, maxBytes)` only тАФ `maxLatencyMs` is
parsed and defaulted but never consulted. `read_batch` returns immediately with whatever is buffered
(it never waits to fill a batch), so a partial batch drains on the engine's next poll. The effective
partial-batch latency is therefore governed by **`delivery.pollIntervalMs`**: a low-rate stream
drains roughly every `pollIntervalMs` regardless of `maxLatencyMs`.
</Aside>

<Aside type="caution" title="compression: zstd is a no-op">
`zstd` is accepted by the schema and the core, but it is treated as `none` тАФ the sink does not apply
compression. Do not rely on compression being applied.
</Aside>

<Aside type="note" title="Numeric fields accept integer-valued floats">
Every numeric `buffer` / `batch` / `delivery` field accepts an integer **or** an integer-valued float
тАФ `1048576` and `1048576.0` are equivalent. Greengrass stores all config numbers as doubles, so
values delivered through `GG_CONFIG` arrive as floats; the core's lenient deserializers accept both,
so the same config parses whether it comes from `GG_CONFIG`, a `FILE`, or a ConfigMap. (A
non-integer float like `1.5` for an integer field is still rejected.)
</Aside>

## Delivery semantics

Streaming is **at-least-once**. The export engine advances the buffer checkpoint past a batch **only**
when every record is acknowledged:

- All records acked тЖТ the checkpoint advances; the batch is durably committed as delivered.
- Partial ack тЖТ only the failed offsets are re-delivered.
- Retryable failure (or a sink callback that throws/raises) тЖТ the **whole batch** is re-delivered and
  the checkpoint is not advanced, so a transient outage cannot lose buffered data.

`delivery.maxRetries` bounds **only** the whole-batch `Failed` path: after that many failed attempts
the engine abandons the batch тАФ it advances the checkpoint past it (so the data is lost), increments
`failedTotal`, and moves on, so one poison batch can't wedge the stream forever. The default `-1`
never gives up. Per-record **partial** failures are *not* bounded by `maxRetries`: the failed offsets
are re-delivered indefinitely (each attempt counts toward `retriesTotal`) until they ack.

Records carry `(partitionKey, timestampMs, payload)`; the engine assigns each one an internal log
**offset**. De-duplicate downstream on `partitionKey` plus that offset тАФ there is no
producer-supplied sequence number.

`flush()` forces the buffer durably **to disk**; it does **not** wait for export to the sink. To
confirm delivery, watch `exportedTotal` / `ackedOffset` in `stats`.

### Buffer-only fallback

A stream with no usable sink still persists appends durably and reports `stats`, but nothing exports.
This happens in Rust when built without `streaming-kinesis` / `streaming-kafka`, and in Java / Python /
TypeScript when a `callback` stream is opened with no sink registered. It is useful for local capture
and tests.

### Reading stats

`stats(name)` returns a snapshot of counters. Field names are camelCase in Java and TypeScript, and
snake_case in Python and Rust.

| Field | Meaning |
|-------|---------|
| `appendedTotal` | Records accepted into the buffer. |
| `exportedTotal` | Records sent to the sink. |
| `droppedTotal` | Records dropped (e.g. `dropOldest`). |
| `retriesTotal` | Export retry attempts. |
| `failedTotal` | Records that failed permanently. |
| `backlog` | Records buffered but not yet exported. |
| `diskBytes` | Bytes currently on disk. |
| `ackedOffset` | Highest acknowledged log offset. |
| `nextOffset` | Next offset to be assigned. |
| `oldestUnackedAgeMs` | Age of the oldest un-acked record. |

Rust's `Stats` carries one extra field, `last_export_error: Option<String>`, which is always
`None` in the wrapper. Java, Python, and TypeScript expose exactly the ten counters above.

## Error handling

The error type mirrors the core's status codes across languages. Java exposes the codes as named
constants; Python and TypeScript carry a numeric `code`; Rust wraps a message string.

| Code | Name | When |
|------|------|------|
| 0 | `OK` | success |
| 1 | `ERR_CONFIG` | invalid streaming config / bad JSON |
| 2 | `ERR_IO` | durable-store I/O error |
| 3 | `ERR_CORRUPT` | corrupt on-disk segment or checkpoint |
| 4 | `ERR_FULL` | ingest queue full under `rejectNew` |
| 5 | `ERR_UNKNOWN_STREAM` | `stream()` / `stats()` for an unconfigured name |
| 6 | `ERR_SINK` | export sink error |
| 7 | `ERR_PANIC` | a Rust panic crossed the FFI boundary |
| 8 | `ERR_INVALID_ARG` | null / invalid native-ABI argument |

<Aside type="note" title="Unconfigured-stream behavior differs in Rust">
Java, Python, and TypeScript **throw** `ERR_UNKNOWN_STREAM` from `stream()` and `stats()` for an
unknown name. In Rust, `stream()` returns `Result::Err` and `stats()` returns `Option::None` (not an
error) when the stream is not configured.
</Aside>
