Every GGCommons component addresses the bus through one **Unified Namespace (UNS)**: a single topic
grammar, a closed set of message **classes**, and a config-driven **identity** stamped on every
envelope. A consumer — a monitoring console, a site historian, an MES bridge — subscribes a small,
uniform set of wildcards and needs **zero per-component knowledge**; every message it receives is
**self-identifying** through its top-level `identity` element, never through topic parsing.

This page is the concept reference the other guides build on; the exact API is in the
[UNS API reference](/reference/uns/).

## The topic grammar

```text
ecv1 / {device} / {component} / {instance} / {class} [ / {channel...} ]
```

| Segment | What it is |
|---------|------------|
| `ecv1` | The fused root literal: EdgeCommons namespace + grammar major version. Grammar bumps are `ecv1 → ecv2`; *payload* versions stay in `header.version`. |
| `{device}` | The physical node — the resolved thing name (the last level of the [hierarchy](#identity-and-the-hierarchy)). |
| `{component}` | The component's sanitized **short name** (the segment after the last `.`). |
| `{instance}` | The logical instance the message pertains to; `main` for component-level messages. |
| `{class}` | One of the eight closed [message classes](#the-eight-message-classes). |
| `{channel...}` | Class-specific sub-path (a metric name, a signal path, a command verb). |

Two deliberate properties:

- **The topic addresses the endpoint, not the hierarchy.** The enterprise's logical hierarchy
  (site/area/line/...) lives in the message's `identity` element, **not** the topic — so topic depth
  is constant regardless of hierarchy depth, and broker-side scoping is by device
  (`ecv1/{device}/#`).
- **Depth-guarded by construction.** The whole grammar stays within AWS IoT Core's 7-slash topic
  limit; the `uns()` builder rejects an over-deep channel at build time instead of letting IoT Core
  silently drop the message.

A multi-site broker deployment can opt in to a site segment after the root with the top-level
`topic.includeRoot: true` config: `ecv1/{site}/{device}/...` (the site is the first hierarchy
value; the setting is a no-op with a single-level hierarchy).

## The eight message classes

| Class | Carries | Publisher | Example tail |
|-------|---------|-----------|--------------|
| `state` | Lifecycle + health + keepalive — **the liveness beacon** | library — **reserved** | `state` |
| `metric` | Operational metrics, incl. the built-in `sys` system measures | library — **reserved** | `metric/signals-ingested` |
| `cfg` | Effective-config snapshot (redacted), at startup + on change | library — **reserved** | `cfg` |
| `log` | Structured log records for remote tail *(class reserved; no built-in publisher)* | library — **reserved** | `log/error` |
| `data` | The telemetry data plane — signal updates, processed outputs | `data()` facade | `data/press12/temperature` |
| `evt` | Events and alarms | `events()` facade | `evt/critical/overtemp` |
| `cmd` | Commands — request/reply, addressed to the **recipient's** inbox | any | `cmd/reload-config` |
| `app` | Arbitrary application pub/sub between components | `app()` facade | `app/order/received` |

The first four are **reserved platform classes**: they are library-owned, and a raw
`messaging().publish(...)` to a reserved-class topic is **rejected** (`ReservedTopicException` /
`ReservedTopicError` / `GgError::ReservedTopic`), so no component can forge another's health or
config. `state` and `cfg` are **leaf** classes (no channel); every other class requires at least one
channel token. `cmd` is the only class whose identity path names the **recipient**; everything else
names the **publisher**.

<Aside type="note" title="What the library publishes for you">
With zero component code, every component is already observable on the UNS:

- **`state`** — the [heartbeat keepalive](/guides/heartbeat/), on by default every 5 s:
  `ecv1/{device}/{component}/main/state` with body `{"status":"RUNNING","uptimeSecs":n}` (and a
  best-effort `{"status":"STOPPED"}` on graceful shutdown).
- **`metric`** — each metric on `ecv1/{device}/{component}/main/metric/{name}` when the
  [metrics](/guides/metrics/) `messaging` target is selected; the heartbeat's system measures arrive
  as a metric named **`sys`**.
- **`cfg`** — the effective (redacted) configuration on `ecv1/{device}/{component}/main/cfg` at
  startup and on every config change.
</Aside>

## The six-wildcard consumer pattern

A consumer's **complete subscription set** is six uniform wildcards — no per-component branches:

```text
ecv1/+/+/+/state        ecv1/+/+/+/cfg        ecv1/+/+/+/evt/#
ecv1/+/+/+/metric/#     ecv1/+/+/+/data/#     ecv1/+/+/+/log/#
```

That covers every component's liveness, config, events, metrics, telemetry, and log tail on
the bus. Group and display by reading each message's `identity` — never by parsing the topic. The
`uns()` API builds these filters for you (`gg.uns().filter(class, scope)`), scoped site-wide or
narrowed to a device/component/instance.

To watch a device's components come alive from any MQTT client:

<Tabs syncKey="os">
<TabItem label="Linux">

```bash
mosquitto_sub -h localhost -p 1883 -t 'ecv1/+/+/+/state' -v
```

</TabItem>
<TabItem label="Windows">

```powershell
mosquitto_sub -h localhost -p 1883 -t 'ecv1/+/+/+/state' -v
```

</TabItem>
</Tabs>

## Identity and the hierarchy

Every envelope built through a config-bound builder carries a top-level **`identity`** element —
a sibling of `header`/`tags`/`body`:

```json
{
  "header":   { "name": "state", "version": "1.0", "timestamp": "…", "uuid": "…", "correlation_id": "…" },
  "identity": {
    "hier": [
      { "level": "site",    "value": "dallas" },
      { "level": "factory", "value": "finishing" },
      { "level": "zone",    "value": "zone-3" },
      { "level": "device",  "value": "gw-01" }
    ],
    "path":      "dallas/finishing/zone-3/gw-01",
    "component": "opcua-adapter",
    "instance":  "kep1"
  },
  "tags": { "app": "line-ctl" },
  "body": { }
}
```

- **`hier`** is the ordered enterprise hierarchy; its **last entry is always the device** (the
  resolved thing name). There is no standalone `device` wire field — the in-memory
  `MessageIdentity` exposes a computed `device` accessor.
- **`path`** is the precomputed `/`-join of the hierarchy values, so consumers group and display
  without re-joining.
- **`component`** / **`instance`** are the addressing suffix — the same tokens as in the topic.
- **`tags` carries business context only** (app, org, cost center). The old `tags.thing` key is
  **removed** — identity is where the device lives now.

The hierarchy is declared once in config — an ordered, freely-named list of levels whose deepest
level is the physical node — and the location values above the device come from the top-level
`identity` config block:

```json
{
  "hierarchy": { "levels": ["site", "factory", "zone", "device"] },
  "identity":  { "site": "dallas", "factory": "finishing", "zone": "zone-3" }
}
```

The last level's value is **always the resolved thing name** (from `-t`, the Kubernetes Downward
API, or `AWS_IOT_THING_NAME`) — putting a `device` key in `identity` is a startup error. With **no**
`hierarchy`/`identity` config at all, the zero-config default is `levels: ["device"]` and the UNS
works out of the box as `ecv1/{thing}/{component}/main/{class}`. Identity resolves **once at
startup** (fail-fast if values don't cover the declared levels), and the library stamps it on every
message. See the [Configuration guide](/guides/configuration/#uns-identity-hierarchy-and-identity).

### Instances are per-message

A component commonly serves many instances (an OPC UA adapter with `kep1`, `plc-2`, ...). The
`{instance}` topic segment and the `identity.instance` field are resolved **per message** through an
instance-scoped handle — `gg.instance("kep1")` — whose `uns()` mints topics and whose message
builder stamps envelopes with that token. Component-level messages default to `main`. See
[`gg.instance()`](/reference/uns/#gginstance--the-instance-scoped-handle).

## Building and validating topics — `gg.uns()`

Never concatenate UNS topics by hand. `gg.uns()` is the topic builder + validator bound to your
component's resolved identity — it enforces the token charset, the class rules, and the ≤ 7-slash
IoT Core depth budget at build time:

<Tabs syncKey="lang">
<TabItem label="Java">

```java
String t = gg.getUns().topic(UnsClass.APP, "order/received");
// -> ecv1/gw-01/my-component/main/app/order/received
gg.getMessaging().publish(t, msg);

// address a peer's command inbox from a received message's identity:
String cmd = gg.getUns().topicFor(peer.getIdentity(), UnsClass.CMD, "reload-config");

// build a consumer filter:
String f = gg.getUns().filter(UnsClass.STATE, UnsScope.all());   // ecv1/+/+/+/state
```

</TabItem>
<TabItem label="Python">

```python
from ggcommons.uns import UnsClass, UnsScope

t = gg.uns().topic(UnsClass.APP, "order/received")
gg.get_messaging().publish(t, msg)

cmd = gg.uns().topic_for(peer.identity, UnsClass.CMD, "reload-config")
f = gg.uns().filter(UnsClass.STATE, UnsScope.all())   # ecv1/+/+/+/state
```

</TabItem>
<TabItem label="Rust">

```rust
use ggcommons::uns::{UnsClass, UnsScope};

let t = gg.uns().topic_with_channel(UnsClass::App, "order/received")?;
gg.messaging()?.publish(&t, &msg).await?;

let cmd = gg.uns().topic_for(peer_identity, UnsClass::Cmd, Some("reload-config"))?;
let f = gg.uns().filter(UnsClass::State, &UnsScope::all())?;   // ecv1/+/+/+/state
```

</TabItem>
<TabItem label="TypeScript">

```typescript
import { UnsClass, UnsScope } from "@edgecommons/ggcommons";

const t = gg.uns().topic(UnsClass.App, "order/received");
await gg.messaging().publish(t, msg);

const cmd = gg.uns().topicFor(peer.identity!, UnsClass.Cmd, "reload-config");
const f = gg.uns().filter(UnsClass.State, UnsScope.all());   // ecv1/+/+/+/state
```

</TabItem>
</Tabs>

`gg.messaging()` takes **literal topics** — it is fully usable for external, non-UNS
topics (bridging a legacy MQTT system, the `cloudwatch/metric/put` contract). The only restriction
is the reserved-class guard above. See the [Messaging guide](/guides/messaging/).

<Aside type="tip" title="Prefer the class-publish facades for data / evt / app">
The example above builds an `app` topic by hand to show `gg.uns()` in the general case — that raw
form is fully supported for any class (the facades add enforcement, not a restriction). But for
the three app-usable classes, reach for [`data()`/`events()`/`app()`](#publishing-telemetry-and-events)
instead of hand-building the topic and body yourself: they mint the same topic, plus construct and
validate the body (the `data` quality guarantee, the `evt` severity-derived channel) that raw
`uns()` + `messaging()` leaves entirely up to you.
</Aside>

## Publishing telemetry and events

For the three app-usable classes, don't hand-build the topic and body with `uns()` + `messaging()` —
use the matching **class-publish facade**: `data()`, `events()`, or `app()`. They mint the same UNS
topic, but also construct and validate the body, so every consumer sees a consistent shape instead of
each publisher re-inventing one (see the
[full API reference](/reference/uns/#class-publish-facades-data-events-app) for every method
signature in all four languages).

<Tabs syncKey="lang">
<TabItem label="Java">

```java
// data(): publish a signal reading — quality defaults to GOOD, serverTs to now
gg.instance("kep1").data().signal("press12/temperature")
    .name("Line1 Temp")
    .addSample(21.5)
    .publish();

// events(): a one-shot informational event, and a stateful alarm pair on the same channel
gg.getEvents().emit("door-open", "front door opened");
gg.getEvents().raiseAlarm("connection-lost", "Modbus link down", null);
gg.getEvents().clearAlarm("connection-lost", null);

// app(): free-form inter-component pub/sub
gg.getApp().publish("OrderReceived", "order/received", body);
```

</TabItem>
<TabItem label="Python">

```python
# data(): publish a signal reading — quality defaults to GOOD, serverTs to now
gg.instance("kep1").data().signal("press12/temperature") \
    .name("Line1 Temp") \
    .add_sample(21.5) \
    .publish()

# events(): a one-shot informational event, and a stateful alarm pair on the same channel
gg.events().emit("door-open", "front door opened")
gg.events().raise_alarm("connection-lost", "Modbus link down")
gg.events().clear_alarm("connection-lost")

# app(): free-form inter-component pub/sub
gg.app().publish("OrderReceived", "order/received", {"orderId": "A-42", "qty": 3})
```

</TabItem>
<TabItem label="Rust">

```rust
// data(): publish a signal reading — quality defaults to Good, server_ts to now
gg.instance("kep1")?.data().publish(
    SignalUpdate::builder().signal_id("press12/temperature").name("Line1 Temp")
        .sample(Sample::new(21.5)).build()
).await?;

// events(): a one-shot informational event, and a stateful alarm pair on the same channel
gg.events().emit_message("door-open", "front door opened").await?;
gg.events().raise_alarm_default("connection-lost", Some("Modbus link down".into()), None).await?;
gg.events().clear_alarm_default("connection-lost", None).await?;

// app(): free-form inter-component pub/sub
gg.app().publish("OrderReceived", "order/received", json!({ "orderId": "A-42", "qty": 3 })).await?;
```

</TabItem>
<TabItem label="TypeScript">

```typescript
// data(): publish a signal reading — quality defaults to Good, serverTs to now
await gg.instance("kep1").data().signal("press12/temperature")
  .name("Line1 Temp")
  .addSample(21.5)
  .publish();

// events(): a one-shot informational event, and a stateful alarm pair on the same channel
await gg.events().emitInfo("door-open", "front door opened");
await gg.events().raiseAlarm("connection-lost", "Modbus link down");
await gg.events().clearAlarm("connection-lost");

// app(): free-form inter-component pub/sub
await gg.app().publish("OrderReceived", "order/received", { orderId: "A-42", qty: 3 });
```

</TabItem>
</Tabs>

Three things worth knowing before you reach for them:

- **The quality default is a guarantee, not a convenience.** `data()` never lets a sample reach the
  bus without a `quality` — if your source has no notion of one, the facade fills `"GOOD"` and marks
  `qualityRaw: "unspecified"`, so a consumer can always distinguish a synthesized verdict from a
  device-reported one. The only thing `data()` ever rejects outright is a missing `signal.id`.
- **The `evt` channel is derived, not chosen.** `events()` builds `evt/{severity}/{type}` from the
  body's own `severity` + `type` fields — you can't publish a body whose topic disagrees with it.
  `raiseAlarm`/`clearAlarm` default `severity` to
  `critical` so the raise and its later clear land on the *same* channel.
- **Channel routing is uniform across all three**: a per-call override, then a config-driven
  `publish.channel` default (instance, then global), then `local`. Only `data()` accepts a
  `stream:<name>` route (for bulk telemetry via [streaming](/guides/streaming/)) — falling back to a
  local publish if no stream is configured rather than dropping the message; `events()`/`app()` are
  local/northbound only, since alarms and app messages are low-rate control-plane traffic.

Raw `uns()` + `messaging()` publishing to `data`/`evt`/`app` is **fully supported** — these classes are
non-reserved, and the facade is the *recommended*, not the *only*, path. Use the raw form for a body
shape the facade genuinely shouldn't own (`data()` even offers a `publishBody()`/`publish_body()`
escape hatch for exactly that — topic + identity handled, body untouched).

## Command addressing and reserved names

- **A component's command inbox** is `ecv1/{device}/{component}/{instance}/cmd/{verb}` — one
  `ecv1/{device}/{me}/+/cmd/#` subscription receives only your own commands, no body-filtering.
  Verbs are lowercase-hyphenated, optionally family-namespaced (`sb/status`).
- **Config fetch** (the `CONFIG_COMPONENT` source): a request to
  `ecv1/{device}/config/main/cmd/get-configuration`, where `config` is a **reserved-by-convention
  logical component name** (the config server is its sole subscriber). Config **push** arrives on
  the target component's own inbox: `ecv1/{device}/{component}/main/cmd/set-config`.
- **Reserved tokens**: the root `ecv1`; the logical component `config`; the pseudo-component
  `_bcast` (device-wide command broadcast, `ecv1/{device}/_bcast/main/cmd/{verb}`); and the whole
  `_`-prefix for system pseudo-components. Don't name components or instances with these.
- **Reply topics are non-UNS** by design: the ephemeral `ggcommons/reply-...` request/reply topics
  are not `ecv1/`-rooted and pass the guard untouched.

## What the library provides

In all four languages (byte-identical topics, structurally identical envelopes, pinned by the
cross-language `uns-test-vectors/` conformance suite):

- The grammar, the eight classes, the reserved-class publish guard, and `gg.uns()` /
  `gg.instance(id)`.
- The top-level `identity` envelope element + the `hierarchy`/`identity`/`topic` config blocks.
- The library-owned publishers: the `state` keepalive ([heartbeat](/guides/heartbeat/)), UNS
  `metric` topics, and the `cfg` effective-config announcements.
- The `request()` internal deadline, MQTT [Last Will](/guides/messaging/#mqtt-last-will-lwt), and
  the `CONFIG_COMPONENT` command remap.
- The `_bcast` `republish-state`/`republish-cfg` broadcast listener every component runs.
- The **`data()`/`events()`/`app()` class-publish facades** — see
  [Publishing telemetry and events](#publishing-telemetry-and-events) above and the
  [full API reference](/reference/uns/#class-publish-facades-data-events-app).
- The **`commands()`** facade (the component command inbox + `ping`/`reload-config`/
  `get-configuration`) in **all four languages** (pinned by `uns-test-vectors/commands.json`).

<Aside type="caution" title="Related components and current limits">
- The **`uns-bridge`** component (repo `edgecommons/uns-bridge`, in the component registry,
  `experimental`) is an envelope-aware bridge per device bus that relays onto a site-wide UNS broker
  (EMQX), with whole-device UNREACHABLE detection and the `republish-state` late-join broadcast; the
  device-side half it uses is the `republish-state`/`republish-cfg` listener every component runs.
- The southbound `sb/*` verbs (`sb/browse`, `sb/read`, confirmed `sb/write`, `sb/subscribe-preview`)
  are **not** a shared, UNS-addressed cross-adapter command family; the underlying capabilities
  (paged address-space browse, a confirmed write with per-entry acknowledgment, regex-matched
  on-demand reads) live in the `opcua-adapter` reference component on that adapter's own topics.
- There is **no `discovery()` or `status()` facade** (for health nuance beyond the automatic
  `state` keepalive) and **no `log`-tail publisher** — use `messaging()` + `uns()` for those.
- Stream records are **not** auto-stamped with identity/hierarchy columns or Hive-partitioned in the
  Parquet/AVRO sink.
</Aside>
