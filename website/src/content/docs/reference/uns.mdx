This is the API reference for the **unified namespace**: the `gg.uns()` topic builder/validator,
the `gg.instance(id)` handle, the `MessageIdentity` type, the normative validation rules, the
reserved-class publish guard, and the `data()`/`events()`/`app()` **class-publish facades** â€” the
sanctioned way to publish the three app-usable UNS classes. For the concept â€” the grammar, the
classes, the six-wildcard consumer pattern â€” read the [Unified Namespace guide](/guides/unified-namespace/)
first; for a task-oriented walkthrough of publishing telemetry/events, see
[Publishing telemetry and events](/guides/unified-namespace/#publishing-telemetry-and-events)
in that guide.

All four languages build **byte-identical topics** and **structurally identical** `identity`
elements; the shared `uns-test-vectors/` conformance suite (topic vectors + golden envelopes) pins
this, and the cross-language interop harness asserts it over a real broker.

## Accessors

| Handle | Java | Python | Rust | TypeScript |
|--------|------|--------|------|------------|
| Topic builder (component-level, instance `main`) | `gg.getUns()` | `gg.uns()` | `gg.uns()` | `gg.uns()` |
| Instance-scoped handle | `gg.instance(id)` | `gg.instance(id)` | `gg.instance(id)` â†’ `Result` | `gg.instance(id)` |

`gg.instance(id)` validates the instance token (charset rule below) and caches the handle per id. It
does **not** require the id to appear in `component.instances[]` â€” instances may be created
dynamically (an unknown id logs a DEBUG diagnostic).

## `UnsClass` â€” the closed class set

Eight classes; `state`/`metric`/`cfg`/`log` are **reserved** (library-owned publish),
`state`/`cfg` are **leaf** (channel forbidden), every other class **requires** â‰Ą 1 channel token.

### Normative validation rules

Identical in all four languages (pinned by `uns-test-vectors/topics.json`):

1. **Token rule** â€” a token is non-empty and contains no `/`, `+`, `#`, `\`, no control characters
   (including the C1 range U+0080â€“U+009F), and no `..` substring. Dots are legal (a full reverse-DNS
   name fits in one level). This is **exactly** the config template sanitizer's blacklist, so any
   sanitized identity value is guaranteed to build a publishable topic.
2. **Depth guard** â€” total `/` count â‰¤ **7** (AWS IoT Core's 8-level limit): channel â‰¤ 3 tokens
   without the root, â‰¤ 2 with `topic.includeRoot: true`. Enforced at **build time**.
3. **Length** â€” total topic â‰¤ **256 UTF-8 bytes**.
4. **Class rules** â€” `state`/`cfg` are leaf (channel forbidden); all other classes require â‰Ą 1
   channel token.
5. **`validate(topic)`** accepts only concrete topics (rejects `+`/`#` â€” use `filter()` for
   subscriptions, whose output is correct by construction).
6. **Root** â€” `topic.includeRoot` inserts the first hierarchy value after `ecv1` **only when the
   hierarchy has â‰Ą 2 levels**; with a single-level hierarchy it is a no-op with a config WARN.

Failures carry a **machine-readable code**, identical across languages:

```text
EMPTY_TOKEN | BAD_CHAR | TRAVERSAL | DEPTH_EXCEEDED | LENGTH_EXCEEDED |
CHANNEL_ON_LEAF | CHANNEL_REQUIRED | BAD_ROOT | BAD_CLASS | WILDCARD_IN_TOPIC
```

| Language | Error type |
|----------|------------|
| Java | `UnsValidationException extends IllegalArgumentException` (`getCode()`) |
| Python | `UnsValidationError(ValueError)` (`.code`) |
| Rust | `EdgeCommonsError::UnsValidation { code: UnsValidationCode, detail }` |
| TypeScript | `UnsValidationError extends Error` (`.code`) |

## `UnsScope` â€” wildcard scopes for `filter()`

An absent field renders as `+`. The `site` position participates only when the builder's effective
root mode is on. Channeled classes get a trailing `/#`; leaf classes end at the class token.

<Tabs syncKey="lang">
<TabItem label="Java">

```java

    public static UnsScope device(String device);
    public static UnsScope component(String device, String component);
```

## `MessageIdentity`

The wire object **and** the component's resolved identity â€” one type. On the wire it is the
envelope's top-level `identity` element (`hier`, `path`, `component`, `instance`); see the
[Unified Namespace guide](/guides/unified-namespace/#identity-and-the-hierarchy) for the JSON shape
and resolution rules.

<Tabs syncKey="lang">
<TabItem label="Java">

```java
    public MessageIdentity(List<HierEntry> hier, String component, String instance);

    public String getPath();                // precomputed '/'-join of the hier values

    public MessageIdentity withInstance(String instance);   // copy with another instance token

    public JsonObject toDict();
```

Deserialize leniency is deliberate: a malformed inbound `identity` yields a null/absent identity
with a warning â€” the message still delivers. `identity` is **optional on the wire**: a message built
without a config-bound builder (the `CONFIG_COMPONENT` bootstrap request, raw bridging of external
systems) legally omits it.

## Class-publish facades: data(), events(), app()

The three **app-usable** UNS classes â€” `data`, `evt`, `app` â€” are non-reserved: any component may
publish them. `data()`/`events()`/`app()` are the **sanctioned publish path**: they go
through the ordinary, guarded `messaging().publish(...)` â€” publishing `data`/`evt`/`app` is not a
privileged operation, so raw `uns()` + `messaging()` publishing (as shown above) is fully
supported â€” but they **construct and validate the body**, fill in the documented defaults, and mint
the topic consistently. Use them; reach for the raw path only when a body is genuinely exotic.

### Accessors

Same convention as `uns()`/`instance()` above: a component-level convenience (instance `main`) and an
instance-scoped form.

| Handle | Java | Python | Rust | TypeScript |
|--------|------|--------|------|------------|
| `data()` (component-level) | `gg.getData()` | `gg.data()` | `gg.data()` | `gg.data()` |
| `events()` (component-level) | `gg.getEvents()` | `gg.events()` | `gg.events()` | `gg.events()` |
| `app()` (component-level) | `gg.getApp()` | `gg.app()` | `gg.app()` | `gg.app()` |
| any of the three, instance-scoped | `gg.instance(id).data()` / `.events()` / `.app()` | same | same (each returns the facade directly; `gg.instance(id)` itself returns `Result`) | same |

### `data()` â€” the telemetry / signal data plane

Publishes `class = data`, `header.name = "SouthboundSignalUpdate"`. It constructs the body
(`device?`, `signal{id, name?, address?}`, `samples[]`) instead of you hand-assembling it, and
sanitizes each `/`-separated token of the signal path into the `data/{channel}` tail â€” the stable
`signal.id` still rides untouched in the body.

<Aside type="tip" title="The quality guarantee">
A sample can **never** reach the bus without a `quality`. If the source doesn't supply one, `data()`
fills `quality: "GOOD"` and marks `qualityRaw: "unspecified"` â€” so a consumer can always tell a
synthesized `GOOD` from a device-reported one. `serverTs` defaults to now; `sourceTs` is **never**
synthesized (left absent when the source has none). The only hard reject is a missing/empty
`signal.id` â€” the one field every consumer keys on.
</Aside>

### `events()` â€” operator events & alarms (evt)

Publishes `class = evt`, `header.name = "evt"`. The channel is **derived from the body**:
`evt/{severity}/{sanitize(type)}` â€” so the topic and the body can never disagree. `timestamp`
defaults to now; `type` is the only required field (it is a channel token).
`raiseAlarm`/`clearAlarm` set `alarm: true` plus `active: true|false` and default `severity` to
`critical`, so a raise and its clear ride the **same** channel â€” a console watching
`evt/critical/#` sees both halves of the alarm lifecycle.

### Channel routing

All three facades resolve their effective channel the same way: **per-call override** (`via(...)` /
`.via(channel)` / the builder's `via`) **â–¸ the config `publish.channel`** (instance-level, then
global) **â–¸ `LOCAL`**. Only `data()` accepts a `stream:<name>` route â€” `events()`/`app()` reject one
at build time, since alarms and app messages are low-rate control-plane, not bulk telemetry.

- **`local`** â€” the default; publishes on the local bus (`messaging().publish`).
- **`northbound`** â€” publishes straight to AWS IoT Core (`messaging().publishToIoTCore`); a
  northbound outage is caught, logged, and never flips local readiness.
- **`stream:<name>`** (`data()` only) â€” serializes the same envelope and appends it to
  `streams().stream(name)` (partition key = `signal.id`, timestamp = the sample's `serverTs`) instead
  of the bus. If no stream is configured, the facade **falls back to a local publish** (with a
  once-only warning) rather than dropping the message or failing the call.

A config-driven default (no code change needed to move a signal from the bus to a stream) is set
under the instance's or the global `component.*` config as `publish: { channel: "..." }` â€” see
`"local"` / `"northbound"` / `"stream:<name>"` above.

## The command inbox: commands()

Where `data()`/`events()`/`app()` publish the outbound app classes, **`commands()`** is the inbound
side of the `cmd` class: a library-owned **command inbox** every component runs on its `main`
instance. It subscribes the component's own command wildcard `ecv1/{device}/{component}/main/cmd/#`
on the PRIMARY (local/IPC) connection and dispatches each incoming `cmd` envelope to a handler by
**verb** â€” the topic channel after `cmd/` (`/`-namespaced verbs included), which the envelope's
`header.name` must equal. It is available in **all four languages**, pinned by
`uns-test-vectors/commands.json`.

### Accessor and registration

| Handle | Java | Python | Rust | TypeScript |
|--------|------|--------|------|------------|
| Command inbox | `gg.getCommands()` | `gg.get_commands()` | `gg.commands()` | `gg.commands()` |

Register a custom verb handler with `register(verb, handler)` in every language; the verb is one or
more `/`-separated `cmd` channel tokens, each validated against the
[token rule](#normative-validation-rules). A verb cannot shadow a built-in or an already-registered
verb.

### Built-in verbs

Three verbs are registered by the library and cannot be shadowed or unregistered:

| Verb | Reply `result` |
|------|----------------|
| `ping` | `{ "status": "RUNNING", "uptimeSecs": n }` â€” liveness/echo (the state keepalive's RUNNING body shape) |
| `reload-config` | `{ "reloaded": true }` â€” re-fetch + re-apply config from the active source |
| `get-configuration` | `{ "config": <redacted effective config> }` â€” the component's own redacted effective config (Flow B) |

### Reply shape

A request carrying `header.reply_to` gets a structured reply on that topic (stamped with the
request's correlation id and the responder's `identity`); a `cmd` without `reply_to` is
fire-and-forget. The reply body is one of:

```json
{ "ok": true,  "result": { /* verb-specific object */ } }
{ "ok": false, "error": { "code": "<CODE>", "message": "<text>" } }
```

An unknown verb replies with `error.code = "UNKNOWN_VERB"`; a malformed or foreign payload (a missing
header, or a `header.name` that does not equal the topic's verb) is ignored at DEBUG â€” never replied
to, never a crash.

## The reserved-class publish guard

Every publish path that takes a client-chosen topic â€” `publish`, `publishRaw`, `publishToIoTCore`,
`publishToIoTCoreRaw`, `request`, `requestFromIoTCore`, and `reply`/`replyToIoTCore` (a hostile
`reply_to` must not turn a responder into a forger) â€” rejects a topic whose class position is a
reserved class (`state` | `metric` | `cfg` | `log`) under the `ecv1` root. `subscribe*` is never
guarded (consumers must read reserved classes), and non-`ecv1` topics (reply topics,
`cloudwatch/metric/put`, legacy/foreign MQTT) pass untouched.

| Language | Error type |
|----------|------------|
| Java | `ReservedTopicException extends IllegalArgumentException` |
| Python | `ReservedTopicError(ValueError)` |
| Rust | `EdgeCommonsError::ReservedTopic(String)` |
| TypeScript | `ReservedTopicError extends Error` |

<Aside type="note" title="The guard is misuse prevention, not a security boundary">
Only Rust's crate-private internal-publish seam is compiler-enforced; in the other three languages
the library's own publishers use a named internal path (`ReservedPublisher` in Java,
`_publish_reserved*` in Python, an `@internal` method in TypeScript) that in-process code could
technically reach. The durable enforcement is per-device **broker ACLs** scoping each component to
its own `ecv1/{device}/#` subtree.
</Aside>
