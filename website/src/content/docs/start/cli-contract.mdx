Every EdgeCommons component selects two things at startup: a **platform** (`--platform` â€” where it
runs) and a messaging **transport** (`--transport` â€” how it talks). A pure precedence *resolver*
maps the command-line flags plus the process environment to one resolved profile, so the same
business logic runs unchanged on Greengrass (IPC), a host or Docker container (dual-MQTT), or
Kubernetes (MQTT).

The contract is **identical in all four languages** â€” the same flags, the same defaults, the same
validation. Only the launcher in front of the flags (`java -jar`, `python3`, the Rust binary,
`node`) differs.

## The six flags

| Flag | Values | Default |
|------|--------|---------|
| `-c` / `--config <SOURCE> [args...]` | `FILE` \| `CONFIGMAP` \| `ENV` \| `GG_CONFIG` \| `SHADOW` \| `CONFIG_COMPONENT` | from the resolved platform profile |
| `--platform <PLATFORM>` | `GREENGRASS` \| `HOST` \| `KUBERNETES` \| `auto` | `auto` (auto-detected) |
| `--transport <TRANSPORT> [path]` | `IPC` \| `MQTT [messaging_config.json]` | derived from the platform |
| `--no-shared-config` | *(takes no value)* | shared layer resolution enabled |
| `-t` / `--thing <name>` | IoT Thing name (full string) | resolved from env (see below) |
| `-h` / `--help` | *(takes no value)* | prints the usage banner and exits 0 |

Platform, transport, and `-c` config-source tokens all parse case-insensitively (each token is
upper-cased before it is matched). `auto` (or an absent `--platform`) lets the resolver auto-detect
the platform from the environment.

### `-c` / `--config` â€” the config source

The first positional argument is the source name; any following positionals are source-specific.
When `--config` is omitted entirely, the source comes from the resolved platform profile
(GREENGRASS â†’ `GG_CONFIG`, HOST â†’ `FILE`, KUBERNETES â†’ `CONFIGMAP`).

| Source | Extra args | Default when args omitted |
|--------|-----------|---------------------------|
| `FILE` | `[path]` | `config.json` |
| `CONFIGMAP` | `[mountDir] [key]` | `/etc/edgecommons` + `config.json` |
| `ENV` | `[var]` | env var `CONFIG` |
| `GG_CONFIG` | `[component] [key]` | key `ComponentConfig` |
| `SHADOW` | `[name]` | the component's shadow |
| `CONFIG_COMPONENT` | *(none)* | â€” |

`FILE` is the host/dev path, `CONFIGMAP` is the Kubernetes-native path (it reads the component
config from a mounted ConfigMap directory with `..data`-swap hot-reload), and `GG_CONFIG` reads from
the Greengrass deployment. See the [Configuration guide](/guides/configuration/) for what each source
reads.

`--no-shared-config` disables shared-layer resolution for this process. It wins over any
`sharedConfig` setting in the component layer and leaves the component layer as the only config
document before validation.

<Aside type="note">
On Kubernetes, the same mounted ConfigMap file can serve as **both** the component config and the
MQTT messaging config â€” under `--transport MQTT` with the `CONFIGMAP` source and no explicit
`--transport MQTT <path>`, the messaging-config path defaults to the resolved ConfigMap file. An
explicit path always wins, and HOST (FILE source) always requires an explicit path.
</Aside>

### `--platform` â€” where it runs

`GREENGRASS` is the on-device, Nucleus-managed path; `HOST` is for bare hosts and Docker;
`KUBERNETES` is the k8s-native profile. Left at `auto`, the resolver auto-detects (first match wins):

1. **GREENGRASS** â€” if env `AWS_GG_NUCLEUS_DOMAIN_SOCKET_FILEPATH_FOR_COMPONENT` or `SVCUID` is set.
2. **KUBERNETES** â€” if the service-account token file
   `/var/run/secrets/kubernetes.io/serviceaccount/token` exists, or env `KUBERNETES_SERVICE_HOST` is set.
3. **HOST** â€” the fallback.

An empty environment value is not treated as a signal, and GREENGRASS wins when both Greengrass and
Kubernetes signals are present. An explicit `--platform` always overrides auto-detection.

### `--transport` â€” how it talks

`IPC` uses Greengrass inter-process communication; `MQTT` is the dual-MQTT provider that connects to
a local broker and to AWS IoT Core at once. When omitted, the transport defaults from the platform
(GREENGRASS â†’ `IPC`, HOST â†’ `MQTT`, KUBERNETES â†’ `MQTT`).

`IPC` is **only valid on GREENGRASS** â€” the resolver rejects `IPC` on HOST or KUBERNETES with a
validation error. For `MQTT`, an optional trailing path supplies the broker/TLS config
(`--transport MQTT ./messaging.json`).

### `-t` / `--thing` â€” the IoT Thing name

Takes the full string value. When `--thing` is absent, the identity resolves in this order:

```text
explicit -t/--thing
  â–¸ (KUBERNETES only) EDGECOMMONS_THING_NAME â–¸ POD_NAME
  â–¸ AWS_IOT_THING_NAME
  â–¸ "NOT_GREENGRASS"
```

Empty environment values count as absent, and the Kubernetes Downward-API tier
(`EDGECOMMONS_THING_NAME` â–¸ `POD_NAME`) only applies when the platform is `KUBERNETES`. The resolver
returns the raw value; sanitization happens later, when the name is interpolated into templates such
as `{ThingName}`.

## Per-platform defaults

Leaving a flag off lets the platform profile fill it in. The CLI-relevant defaults:

| Default | GREENGRASS | HOST | KUBERNETES |
|---------|-----------|------|------------|
| transport | `IPC` | `MQTT` | `MQTT` |
| config source | `GG_CONFIG` | `FILE` | `CONFIGMAP` |

Each profile also carries downstream subsystem defaults â€” logging format, metric target,
metric-log path, credentials key provider, and health-endpoint enablement â€” applied by the
respective subsystems rather than by the CLI resolver:

| Profile default | GREENGRASS | HOST | KUBERNETES |
|-----------------|-----------|------|------------|
| logging format | library default | library default | `json` (stdout-JSON) |
| metric target | library `log` | library `log` | `prometheus` |
| credentials key provider | library `file` | library `file` | `env` (base64 KEK) |
| health endpoint | off | off | on |

The effective precedence is `explicit flag â–¸ explicit config value â–¸ platform-profile default â–¸ library default`.
The runtime axes (transport, config source, identity) are resolved from the **flag** tier by the
resolver; the subsystem settings above honour an **explicit config value** first, then fall back to
the profile default at each subsystem's init site.

## How the flags reach your code

You rarely call the parser directly â€” the builder parses the contract for you. Hand it the
process arguments and read the resolved services back off the built instance.

<Tabs syncKey="lang">
<TabItem label="Java">

```java
import com.mbreissi.edgecommons.EdgeCommons;
import com.mbreissi.edgecommons.EdgeCommonsBuilder;

public final class App {
    public static void main(String[] args) throws Exception {
        EdgeCommons gg = EdgeCommonsBuilder.create("com.example.MyComponent")
                .withArgs(args)   // parses --platform/--transport/-c/-t
                .build();

        var messaging = gg.getMessaging();
        // gg.getConfigManager(), gg.getMetrics(), gg.getStreams(),
        // gg.getCredentials(), gg.getParameters() are also available.
    }
}
```

</TabItem>
<TabItem label="Python">

```python
import sys
from edgecommons.edgecommons_builder import EdgeCommonsBuilder

gg = (
    EdgeCommonsBuilder.create("com.example.MyComponent")
    .with_args(sys.argv[1:])   # args WITHOUT the program name
    .build()
)

messaging = gg.get_messaging()
# gg.get_streams(), gg.get_credentials(), gg.get_parameters() are also available.
```

</TabItem>
<TabItem label="Rust">

```rust
use edgecommons::EdgeCommonsBuilder;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    // Rust takes the FULL argv, including the program name at index 0.
    let argv: Vec<String> = std::env::args().collect();

    let gg = EdgeCommonsBuilder::new("com.example.MyComponent")
        .args(argv)
        .build()
        .await?;

    let _args = gg.args();          // the resolved ParsedArgs
    let _messaging = gg.messaging()?; // returns Result
    // gg.config(), gg.streams() are also available.
    Ok(())
}
```

</TabItem>
<TabItem label="TypeScript">

```typescript
import { EdgeCommonsBuilder } from "@edgecommons/edgecommons";

// Supply argv WITHOUT the node/script prefix (process.argv.slice(2)).
const gg = await new EdgeCommonsBuilder("com.example.MyComponent")
  .args(process.argv.slice(2))
  .build();

const parsed = gg.args();       // the resolved ParsedArgs
const messaging = gg.messaging();
```

</TabItem>
</Tabs>

<Aside type="caution" title="argv conventions differ">
**Rust** parses the *full* argv including the program name at index 0 (C-style). **Java**
(`main`'s `String[] args`), **Python** (`sys.argv[1:]`), and **TypeScript**
(`process.argv.slice(2)`) all expect the arguments *without* the program name. Pass the wrong slice
and the first real flag is silently dropped or mis-parsed.
</Aside>

## Full command lines

The flags are identical across languages; only the launcher changes. A HOST run with the dual-MQTT
transport, a local config file, and an explicit thing name:

<Tabs syncKey="os">

```bash

  --platform HOST --transport MQTT ./messaging.json \
  -c FILE ./config.json -t my-thing

# Python
python3 main.py \
  --platform HOST --transport MQTT ./messaging.json \
  -c FILE ./config.json -t my-thing

# Rust
./target/release/my-component \
  --platform HOST --transport MQTT ./messaging.json \
  -c FILE ./config.json -t my-thing

# TypeScript

  --platform HOST --transport MQTT ./messaging.json \
```

## Migrating from `-m` / `--mode`

The `-m` / `--mode` flag is not supported; use the two orthogonal flags `--platform` and
`--transport` instead. All four SDKs reject `-m`/`--mode` with the same guidance:

```text
The -m/--mode flag has been removed. Use --platform GREENGRASS|HOST|KUBERNETES and
--transport IPC|MQTT instead (e.g. '-m STANDALONE <path>' becomes
'--platform HOST --transport MQTT <path>').
```

Typical translations:

| Old | New |
|-----|-----|
| `-m STANDALONE ./messaging.json` | `--platform HOST --transport MQTT ./messaging.json` |
| `-m GREENGRASS` | `--platform GREENGRASS` (transport defaults to `IPC`) |

<Aside type="note" title="Rejection breadth differs">
**Java** and **Python** reject any token that starts with `-m` or `--mode=` (so even an attached
form like `-mSTANDALONE` is caught). **Rust** and **TypeScript** reject only the exact tokens `-m`
and `--mode`, so a custom flag that merely begins with `-m` is left alone there.
</Aside>

## Cross-language notes

The contract is identical, but a few details of the *parsed result* differ between SDKs â€” relevant
only if you inspect the parsed args directly rather than just reading services off the builder.

- **`thing` vs resolved identity.** Rust's `ParsedArgs` exposes both `thing: Option<String>` (the
  raw `-t` value, verbatim) and `identity: String` (the resolved name). TypeScript's
  `ParsedArgs.thing` is the **resolved** identity (never undefined) â€” there is no separate raw field.
  Java's `ParsedCommandLine.thingName` is likewise the resolved identity. Do not assume `thing` means
  the raw flag everywhere.
- **Where the messaging-config path lives.** Java threads the MQTT messaging-config path through the
  resolver (its `ResolverInputs` / `ResolvedProfile` carry it). Python, Rust, and TypeScript keep
  those resolver types at four fields and compute the ConfigMap-default messaging path in the CLI
  layer instead. Either way the resolved path is on the parsed result.
- **Config-source typing.** Rust and TypeScript parse `-c` into a typed `ConfigSourceSpec`
  (a Rust enum / a TS discriminated union, both including `CONFIGMAP`). Python validates the token
  against its `ConfigSource` enum. Java passes the raw `configArgs` array straight through to the
  config-manager builder.
- **Enum casing (Rust).** Rust uses `Platform::{Greengrass, Host, Kubernetes}` and
  `Transport::{Ipc, Mqtt}` (PascalCase) internally; Java, Python, and TypeScript use the uppercase
  token names. All four parse the CLI token case-insensitively, so this never affects the command line.
- **Metric-log filename.** The default local metric-log path is `./logs/{ComponentFullName}.metric.log`
  in Java, Rust, and TypeScript, but `./logs/{ComponentFullName}_metric.log` (underscore) in Python â€”
  each library matches its own default filename suffix.
- **Public parser entry point.** Java `EdgeCommons.processArgs`, Rust `cli::parse_from`, and TS
  `parseArgs` are public; Python's `_process_args` is private â€” the public Python path is the builder
  (`with_args`) only.
