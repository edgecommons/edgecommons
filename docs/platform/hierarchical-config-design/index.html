Implementation design grounded in the existing layered-config code

Make shared config follow the user's real hierarchy.

The previous two-layer model proved the right merge contract: raw partial layers become one validated effective config. The next step is to stop treating "shared" as exactly one line layer, and instead resolve every catalog ancestor the deployment defines: enterprise, site, building, zone, line, and then the component leaf. The device remains runtime identity.

            Keep
            existing config providers, merge semantics, validation timing, hot reload, and one effective snapshot.
          

          

            Change
            the ConfigComponent catalog from "one base plus one component" to "ordered lineage layers."

Current implementation

            
What exists already is the right foundation.

            

              The first layered-config implementation was deliberately conservative. A provider loads a
              component layer, an optional shared layer is resolved from the same provider family,
              raw control fields are stripped, and the merged result is validated as the only runtime
              config document.
            

            

              The arbitrary-depth UNS identity model already exists: `hierarchy.levels` can describe
              a real hierarchy, and `identity` supplies values for every level above the device. The
              limitation is not the identity model. The limitation is the config-layer resolver.
            

          

          

            

              Java
              
`LayeredConfigCoordinator`

              
Builds a candidate from component plus base, validates the effective result, and is the startup seam to replace with N-layer merge.

            

            

              TS / Rust / Python
              
Parallel merge helpers

              
All four implementations use ordered deep merge semantics: objects merge, arrays and scalars replace, later layers win.

            

            

              ConfigComponent
              
Catalog bundles

              
The previous server returned `{ "base": ..., "component": ... }`; the target server resolves catalog lineage and returns ordered `layers[]`.

            

            

              Runtime
              
One effective config

              
`gg.config()` and the library-owned `cfg` publisher expose the merged, redacted effective config, not raw layers.

            

          

        


        

          

            
Target model

            
Shared config becomes ancestry.

            

              A component does not inherit from "shared" as a special second document. It inherits
              from a deterministic chain of parent scopes served by ConfigComponent. The chain can
              be a single component layer or a rich enterprise/site/building/zone/line/component
              lineage with runtime device identity, but workload components still read one config source.

Line deployment

                

                

              

            

          


          

            

              
Single parent chain

              

                The target model allows arbitrary depth but still resolves one lineage per component.
                Multiple inheritance is left out because it makes operator intent and conflict
                reporting much harder.
              

            

            

              
Central lineage resolver

              

                Arbitrary ancestry is resolved by ConfigComponent. FILE, ENV, and CONFIGMAP are
                catalog source descriptors; GG_CONFIG and SHADOW can deliver the server bootstrap
                config. None of them becomes a workload-side graph resolver.
              

            

            

              
Merge after resolution

              

                Each raw layer can remain partial. Only the final effective document is schema
                validated and exposed to application code.
              

            

          

        


        

          

            
Layer contract

            
A small internal contract replaces the two-layer assumption.

            

              The public component config shape should stay focused on effective runtime behavior.
              Hierarchy metadata lives in the ConfigComponent catalog. The `CONFIG_COMPONENT` source
              receives `ResolvedConfigLayer[]`, ordered from broadest ancestor to component, and the
              existing merge function consumes the same list shape it was already built to support.
            

          

          

            
type ResolvedConfigLayer = {
  id: string
  kind: "scope" | "component"
  scope?: Record<string, string> // required for scope layers
  component?: string              // required for final layer
  config: JsonObject
}

effective = deepMerge(
  lineage.map(layer => layer.config)
)

          

        


        

          

            
Pure config fragments

            

              Each layer carries metadata outside `config`. The `config` object is a partial
              EdgeCommons config fragment. New public authoring should not use `extends`,
              `sharedConfig`, or `base` inside component config.
            

          

          

            
Catalog resolver

            

              Use `lineage_for()` for request/reply and `lineages()` for pushes. Follow catalog `parent` links until root,
              detect cycles, enforce a practical maximum depth, and preserve source provenance for
              diagnostics.
            

          

          

            
Identity consistency

            

              Lower layers may add missing lower-level identity keys but must not change an
              already-owned ancestor identity value. Moving a component to a different site, zone,
              or line is an ancestry change, not a local override.
            

          

          

            
Atomic reload

            

              A reload re-resolves the full lineage. If any layer is missing, malformed, cyclic, or
              produces an invalid effective config, the previous effective snapshot stays active.
            

          

        


        

          

            
Provider matrix

            
Every provider participates without every provider becoming recursive.

            

              To minimize the core extension surface, arbitrary hierarchy is resolved by
              ConfigComponent. Other providers back the ConfigComponent catalog or continue to load
              a single effective document directly. Click a provider to inspect its role.
            

          

          

            

              
              
              
              
              
              
            

            

          

        


        

          

            
Deployment fit

            
ConfigComponent is the common control plane; transport is the platform detail.

            

              The optimal deployment shape is the same on every platform: ordinary components read
              config from ConfigComponent and receive ordered `layers[]` bundles. The platform decides
              how that request/reply and push path travels: local MQTT on hosts, in-cluster MQTT on
              Kubernetes, and Greengrass IPC on Nucleus-managed devices.
            

          

          

            
            
            
          

          

        


        

          

            
Architect handoff

            
Implementation should replace the two-layer resolver, not broaden every provider.

            

              The development plan should keep the runtime contract narrow: ConfigComponent owns
              catalog lineage, core libraries own parsing and merging `layers[]`, and application code
              still observes one validated effective config snapshot.
            

          


          

            

              
Non-negotiable success criteria

              
    
                
  • Ordinary components use `-c CONFIG_COMPONENT` for hierarchical config on HOST, Kubernetes, and Greengrass.
    • 
                
  • ConfigComponent responses and pushes contain only ordered `layers[]` bundles, never `{ base, component }`.
    • 
                
  • Clients merge `layers[].config`, validate the effective config, and keep reject-and-keep reload behavior.
    • 
                
  • FILE, ENV, and CONFIGMAP are supported catalog sources; GG_CONFIG and SHADOW can deliver ConfigComponent bootstrap config.
    • 
                
  • `extends`, `sharedConfig`, `base`, and `--no-shared-config` are removed from the public authoring model.
    • 
              

            

            

              
Minimal replacement boundary

              

                Do not teach every ordinary config provider to follow parent chains. That multiplies
                platform-specific edge cases and puts hierarchy resolution inside every workload.
                The replacement boundary is:
              

              
    
                
  1. ConfigComponent catalog source adapters load a catalog snapshot.
    2. 
                
  2. ConfigComponent resolves a component token to root-to-leaf layer metadata.
    3. 
                
  3. Language libraries consume the `layers[]` protocol and deep-merge configs.
    4. 
              

            

          


          

            

              
              
              
              
              
            

            

          

        


        

          

            
Catalog and wire contract

            
The server catalog is a graph; the client protocol is an ordered list.

            

              The catalog can express arbitrary hierarchy, but the wire protocol sent to workload
              components should be deliberately boring: a catalog version, optional provenance, and
              ordered layers with pure config fragments.

Resolution algorithm

              
    
                
  1. Accept `GetConfiguration` body `{ "component": string }` and sanitize to the short component token.
    2. 
                
  2. Load the active catalog snapshot; reject if none is valid.
    3. 
                
  3. Find `components[token]`; reject unknown tokens with `CONFIG_NOT_FOUND`.
    4. 
                
  4. Walk `parent` links through `nodes`, collecting leaf-to-root references.
    5. 
                
  5. Reject missing parents, duplicate node IDs, cycles, depth above 64, and malformed scopes.
    6. 
                
  6. Reverse to root-to-leaf order, append the component layer, and return `layers[]`.
    7. 
                
  7. On catalog reload, regenerate pushes for every component from the newly promoted catalog.
    8. 
              

            

            

              
Client apply algorithm

              
    
                
  1. Parse a reply or push as a lineage bundle. Reject absent or empty `layers[]`.
    2. 
                
  2. For each layer, require object metadata and object-valued `config`.
    3. 
                
  3. Validate monotonic scope ownership and identity ownership before merge.
    4. 
                
  4. Deep-merge `layers[].config` using the current object-merge, array-replace, scalar-replace semantics.
    5. 
                
  5. Validate only the merged effective config against the existing schema.
    6. 
                
  6. On reload failure, log the lineage error and retain the previous effective snapshot.
    7. 
              

            

          

        


        

          

            
Runtime pipeline

            
The runtime still hands application code one config.

            

              The implementation change sits before `ConfigManager` or `Config.fromValue`. Once the
              effective config is accepted, all existing subsystems keep their current contracts.
            

          

          

            

              1
              Request lineage
              
Client asks ConfigComponent for its token.

            

            

              2
              Receive layers
              
Server returns root-to-component `layers[]`.

            

            

              3
              Merge ordered layers
              
Objects merge, arrays/scalars replace.

            

            

              4
              Validate effective config
              
Reject-and-keep on bad reloads.

            

            

              5
              Publish one snapshot
              
`gg.config()` and `cfg` stay effective-only.

            

          

        


        

          

            
Operator experience

            
The design should feel safer than copy-paste.

            

              The user benefit is not just less duplication. Operators need to see which layer set a
              value, understand why a reload was rejected, and preview the effective config before a
              rollout hits production.
            

          

          

            

              Explain
              
`edgecommons config explain` should print lineage, source URI, version/hash, and winning layer per path.

            

            

              Lint
              
Detect cycles, missing parents, ConfigMap projection artifacts, identity conflicts, and untemplated shared stream paths.

            

            

              Preview
              
ConfigComponent can expose a dry-run endpoint that returns the resolved lineage and effective config for a component token.

            

            

              Observe
              
Startup and reload logs should name the lineage without dumping config values. Redacted `cfg` remains the effective contract.

            

          

        


        

          

            
Cutover

            
Replace the two-layer shape with fewer concepts.

          

          

            

              
              
Component leaf

              
A component catalog entry is always the leaf. If it declares no parent, the lineage is simply `[component]`.

            

            

              
              
Catalog parent

              
`parent` means parent scope in the ConfigComponent catalog. There is no special "base" concept.

            

            

              
              
Layer array

              
ConfigComponent replies and pushes ordered `layers[]` bundles only. Clients merge that array directly.

            

            

              
              
Docs and vectors

              
Conformance tests and public docs use catalog, lineage, layer, and effective config terms consistently.

              conformance vectors, local provider smokes, interop, and deployed regression.

Recommended decisions

            
Make the next implementation narrow, deep, and reversible.

          

          

            

              Decision
              Recommendation
              Reason
            

            

              Depth
              Arbitrary-depth single parent chain
              Matches enterprise/site/building/zone/line without DAG ambiguity.
            

            

              Merge
              Keep current deep merge unchanged
              Existing vectors and mental model stay valid; only the resolver grows.
            

            

              Provider scope
              Centralize hierarchy in ConfigComponent
              Provider-specific complexity stays in catalog source adapters, not every workload.
            

            

              ConfigComponent
              Replace `base/components` bundles with catalog `nodes/components` and `layers[]`
              Server can pre-resolve catalog lineage but clients still perform the merge.
            

            

              Public API
              Remove `extends`, `sharedConfig`, `base`, and `--no-shared-config`
              Breaking changes are allowed, so stale two-layer authoring concepts should not survive.
            

            

              Diagnostics
              Add explain/lint before new authoring docs are promoted
              N-layer inheritance without lineage visibility will be hard to operate safely.
