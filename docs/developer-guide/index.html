GGCommons Developer Guide

GGCommons is a set of libraries for building AWS IoT Greengrass v2 components. It bundles the cross-cutting concerns every edge component needs — configuration, messaging, metrics, heartbeat, logging, credentials, parameters, and telemetry streaming — so you write only business logic.

The same library ships in four languages — Java Python Rust TypeScript — as deliberate mirrors of one another: the same config schema, the same command-line contract, the same subsystem boundaries, and the same on-the-wire message format. Pick your language with the switcher in the top bar and every code sample on this page follows it.

Who this guide is for

Component developers building on top of GGCommons. It does not cover contributing to the libraries themselves — for that, see each library's own README and CLAUDE.md.

What the library gives you

One runtime, two modes

The same business logic runs under Greengrass IPC or as a standalone container against MQTT. Selected at startup.

Unified messaging

Publish/subscribe and request/reply over one interface, whether the transport is Greengrass IPC or dual MQTT.

Config that resolves itself

Five config sources, template variables, multi-instance, hot reload, and JSON-schema validation.

Observability built in

Metrics (CloudWatch EMF and more) and automatic heartbeats with no boilerplate.

Edge-grade secrets

An encrypted, offline-first local vault synced from AWS Secrets Manager.

Durable telemetry egress

Store-and-forward streaming to Kinesis and Kafka with an embedded persistent buffer.

Getting started

The fastest path to a running component is to scaffold one with the GGCommons CLI, which copies a language template and substitutes your component name.

1. Install the CLI and scaffold a component

pipx install ./cli                      # or: python -m pip install ./cli
ggcommons doctor                        # check prerequisites (git, gdk, mvn, cargo, python3, aws)

# scaffold ./MyComponent for your chosen language
ggcommons create-component -n com.example.MyComponent -l PYTHON   # JAVA | PYTHON | RUST | TYPESCRIPT

2. Initialize the framework in your entry point

Every component constructs a GGCommons runtime from the command-line arguments using the fluent builder, pulls the services it needs, runs, and shuts down cleanly.

3. Run it locally

Bring up a local MQTT broker (standing in for Greengrass IPC), then run the component in STANDALONE mode:

# start the shared broker (EMQX)
docker compose -f test-infra/compose.yaml up -d

# Python
python3 main.py -m STANDALONE standalone-messaging.json -c FILE config.json -t my-thing
# Java
java -jar target/<artifact>.jar -m STANDALONE ./standalone-messaging.json -c FILE ./config.json -t my-thing
# Rust
cargo run -- -m STANDALONE ./standalone-messaging.json -c FILE ./config.json -t my-thing
# TypeScript
node dist/main.js -m STANDALONE ./standalone-messaging.json -c FILE ./config.json -t my-thing
Tip

Subscribe to heartbeat/+/+ with an MQTT client (such as MQTTX) to watch your component's heartbeats as soon as it starts. Packaging and deployment to a real core use the GDK: gdk component build then gdk component publish.

Runtime modes

Every component runs in one of two modes, selected at startup with -m/--mode. Most of the architecture exists to abstract this difference away — the same business logic runs in both.

ModeTransportConfig sourceBest for
GREENGRASS (default)Greengrass IPCGreengrass deploymentOn-device, Nucleus-managed production
STANDALONEDual MQTT (local broker + AWS IoT Core)File / env / shadowKubernetes, Docker, bare hosts, local dev
Important

STANDALONE requires a messaging-config JSON path: -m STANDALONE <messaging_config.json>. Passing STANDALONE without a path is a hard error. See Messaging for the file format.

Command-line contract

All four languages accept the same three core options. Components may add their own options on top.

OptionValuesDefaultPurpose
-c, --config FILE · ENV · GG_CONFIG · SHADOW · CONFIG_COMPONENT GG_CONFIGWhere configuration is loaded from
-m, --mode GREENGRASS · STANDALONE <path> GREENGRASSRuntime mode and messaging transport
-t, --thing any IoT Thing name—The device identity this component runs as
-h, --help——Print usage and exit

Config-source arguments

The -c option takes a source keyword followed by optional positional arguments:

-c FILE [file_path]                      # JSON file (default: ./ in the working dir)
-c ENV [env_var_name]                    # JSON in an env var (default: GGCOMMONS_CONFIG)
-c GG_CONFIG [component_name] [key]       # Greengrass deployment config (default; current component)
-c SHADOW [shadow_name]                   # AWS IoT Device Shadow desired-state
-c CONFIG_COMPONENT                       # a dedicated configuration-management component
Note

Always pass the full component name to the builder (matching recipe.yaml). The {ComponentFullName} template then resolves to the Greengrass-managed work directory; a short name resolves to a path the component cannot create, and vault/stream directories will fail.

Builders

Objects in GGCommons are constructed with fluent builders rather than raw constructors. Builders validate required parameters, let you set only what you need, and stay backward-compatible as new options are added. The three you will use most are GGCommonsBuilder (the runtime), MessageBuilder (messages), and MetricBuilder (metrics).

GGCommonsBuilder

Creates and initializes the runtime. create()/new takes the full component name; the remaining methods are optional.

MessageBuilder

Builds a message from a name, version, and payload. withConfig stamps the message with the tags and identity from your configuration.

MetricBuilder

Defines a metric with named measures (each with a unit and storage resolution) and dimensions. Component and Thing name dimensions are added automatically.

Note

In Java the legacy new GGCommons(...) / new Metric(...) constructors still work for backward compatibility but are deprecated in favor of the builders. Rust and TypeScript are builder-only.

Configuration

The configuration subsystem loads your component's settings from one of several sources, resolves template variables, validates against a JSON schema, and notifies you when configuration changes at runtime.

Structure

A configuration document has framework-managed sections and one application section:

  • logging, heartbeat, metricEmission, tags — consumed by the framework.
  • component.global — your settings, shared across all instances.
  • component.instances — an array of per-instance settings for multi-instance components.

Template variables

Any configuration string may reference variables that are resolved automatically:

VariableResolves to
{ComponentName}Short component name, e.g. MyComponent
{ComponentFullName}Full name with version, e.g. com.example.MyComponent-1.0.0
{ThingName}The IoT Thing name from -t
{InstanceId}Current instance id (within instances)
{anyTag}Any key defined under tags, e.g. {site}

Example document

{
  "logging": { "level": "INFO" },
  "heartbeat": { "intervalSecs": 30, "targets": [ { "type": "metric" } ] },
  "tags": { "site": "factory-1", "region": "us-east-1" },
  "component": {
    "global": {
      "serverUrl": "https://api.{region}.example.com",
      "publish_interval": 5
    },
    "instances": [
      {
        "id": "line-1",
        "publishTopic": "{ThingName}/{ComponentName}/{InstanceId}/data"
      }
    ]
  }
}

Reading configuration in code

Reacting to configuration changes

Register a change listener to reload settings without restarting the component (for example when a Greengrass deployment or shadow update arrives).

Validation

Configuration is validated against ggcommons-config-schema.json at startup. Point your editor at that schema for auto-complete and inline error highlighting while you write config files.

Messaging

Messaging is one interface over two providers. In GREENGRASS mode it uses Greengrass IPC; in STANDALONE mode it uses a dual-MQTT provider that connects to a local broker and, optionally, AWS IoT Core at the same time. Connections and subscriptions block until confirmed, so once a call returns you know it is live.

Messages

A message has a header (name, version, correlation id, timestamp, reply-to), tags (contextual metadata from config or added at runtime), and a body (your payload). Build them with MessageBuilder.

Publish / subscribe

Request / reply

Request/reply uses correlation ids under the hood. The requester gets a future/promise that resolves with the reply; the responder calls reply with the original message.

STANDALONE messaging configuration

In standalone mode, the file passed to -m STANDALONE <file> defines both MQTT connections. The local broker can use username/password or mutual TLS; IoT Core uses certificate-based auth.

{
  "messaging": {
    "local": {
      "host": "mqtt-broker.local",
      "port": 1883,
      "clientId": "my-component-local",
      "credentials": { "username": "mqtt-user", "password": "mqtt-password" }
    },
    "iotCore": {
      "endpoint": "your-endpoint.iot.us-east-1.amazonaws.com",
      "port": 8883,
      "clientId": "my-component-iotcore",
      "credentials": {
        "certPath": "/certs/device-cert.pem",
        "keyPath":  "/certs/private-key.pem",
        "caPath":   "/certs/root-ca.pem"
      }
    }
  }
}
Same code, different deployment

Your publish/subscribe/request/reply code is identical in both modes. Only the mode flag and this config change when you move a component between Greengrass and a container.

Metrics

The metrics subsystem emits measurements in Embedded Metric Format (EMF) for CloudWatch compatibility. You define a metric once (its measures and dimensions), then emit values against it. Where those metrics go is chosen by configuration, not code.

Define and emit

Targets

TargetBehaviorUse when
cloudwatchDirect to CloudWatch, batched (≤20/call) on an intervalLow volume, or you need a custom batching interval
cloudwatchcomponentVia the Greengrass CloudWatch component over IPCLegacy — no longer recommended
logLocal rotating file in EMFLocal debugging, or paired with the LogManager component
messagingPublishes EMF over IPC or to IoT CoreReal-time monitoring or custom processing

Configuration

{
  "metricEmission": {
    "target": "cloudwatch",
    "namespace": "Production/MyApp",
    "targetConfig": { "intervalSecs": 30 },
    "largeFleetWorkaround": true
  }
}

Set largeFleetWorkaround: true on large deployments to aggregate the coreName dimension into ALL and cut CloudWatch cost. The default namespace is ggcommons.

Heartbeat

Heartbeats give you liveness and resource-utilization signals with zero code. They start automatically at initialization and emit on a schedule. You configure what they measure and where they go — everything else is handled for you.

What it measures

MeasureKeyDefault
CPU usage (%)cpuon
Memory usage (MB)memoryon
Disk usage (%)diskoff
Thread countthreadsoff
Open file countfilesoff
File descriptorsfdsoff

Availability of individual measures varies by language and platform.

Targets

Heartbeats route through either the metric subsystem (no extra config) or the messaging subsystem (to a topic, IPC or cloud). You can list multiple targets.

{
  "heartbeat": {
    "intervalSecs": 30,
    "measures": { "cpu": true, "memory": true, "threads": true, "files": false },
    "targets": [
      { "type": "metric" },
      { "type": "messaging",
        "config": { "destination": "ipc",
                    "topic": "ggcommons/{ThingName}/{ComponentName}/heartbeat" } }
    ]
  }
}
Tip

The default messaging topic is ggcommons/{ThingName}/{ComponentName}/heartbeat. Subscribe to heartbeat/+/+ (or your custom topic) during local development to confirm the component is alive. The minimum interval is 1 second; the default is 5.

Logging

Logging is configured through the logging section — root level, message format, optional file output, and per-logger overrides. Log file paths support the same template variables as the rest of the system.

Options

KeyPurposeDefault
levelRoot level: TRACE…FATALINFO
java_format / python_format / rust_format / ts_formatPer-language log message pattern (each library reads only its own key; the others ignore it)timestamped pattern
fileLoggingEnable file outputfalse
logFilePathFile path (templated)—
loggersPer-logger level overrides—
globalControlTake full control of the logging systemfalse
 
{
  "logging": {
    "level": "INFO",
    "fileLogging": true,
    "logFilePath": "/greengrass/v2/logs/{ComponentName}-{ThingName}.log",
    "loggers": {
      "com.breissinger.ggcommons.messaging": "INFO",
      "com.breissinger.ggcommons.metrics": "WARN"
    }
  }
}

Once initialized, write logs with your language's normal logging facility — the framework configures the underlying logger for you.

Note

Enable globalControl when you want GGCommons to own the logging configuration entirely (replacing any external config) and reconfigure it dynamically on config change. Leave it off to keep your existing logging setup and only apply component-specific levels.

Credentials & vault

The credentials subsystem is an encrypted, offline-first local vault on the device. It stores named, versioned secrets encrypted at rest and can be seeded and refreshed from a central source (AWS Secrets Manager first). Reads are served entirely from the local vault, so your component keeps working when the cloud is unreachable — essential at the edge.

Reading a secret

Obtain the service with gg.credentials() (it is null/None/Option when there is no credentials config section). Keys are automatically namespaced by Thing and component, so you reference short names. Never log a secret value.

Edge security model

A shared device vault means every component on the device that can unlock it can read every secret in it. Enforce least privilege on the sync side (the device only pulls the secrets its IAM/TES role entitles it to). Secret values are redacted in diagnostics and must never be written to logs.

Parameters

Parameters give you offline-capable, source-agnostic access to externalized configuration — endpoints, feature flags, tuning values. It is a peer of credentials but aimed at configuration rather than secrets, and its signature operation is fetching a whole subtree by path.

Pluggable sources

Source (type)BackendTypical mode
awsSsmAWS SSM Parameter Store (with decryption)GREENGRASS (via TES)
mountedDirA directory tree — K8s ConfigMap/Secret mounts, Docker secretsSTANDALONE / Kubernetes
envEnvironment variables under a prefixSTANDALONE / containers
customYour own ParameterSource (HashiCorp, Azure, …)any

The service above the source is offline-first: reads always come from a cache and never make a network call; a failed background refresh keeps serving last-known values. For remote sources the cache is persisted and encrypted at rest (reusing the vault crypto); for already-local sources it stays in memory.

Reading parameters

Configuration

parameters:
  source:
    type: "awsSsm"          # awsSsm | mountedDir | env | custom
    region: "us-east-1"
    withDecryption: true
    # mountedDir:  root: "/etc/config"   securePaths: [ "/etc/secrets" ]
    # env:         prefix: "MYAPP_"
  sync:
    names: [ "/skeleton/region", "/skeleton/poolSize" ]   # selective bootstrap
  refreshIntervalSecs: 300
Credentials vs parameters

Use gg.credentials() for secrets that need rotation and vault-grade handling. Use gg.parameters() for configuration values (with secure values as one option). Components routinely use both at once.

Telemetry streaming

Streaming is the high-throughput, store-and-forward data plane: one-way, high-rate, durable, batched egress to Kinesis and Kafka. It is built around a portable, embedded, persistent buffer so it keeps accepting records during disconnection. It is opt-in (active only when your config has a streaming section) and does not change any existing API.

Appending records

Get a named stream from gg.streams() (in Java/Python/TS this is null/None/undefined when there is no streaming config; in Rust the service is always returned and stream(name) errors if that stream isn't configured). Append a payload with a partition key (drives ordering and downstream sharding — partition by Thing for ordered per-device delivery) and a timestamp.

Configuration

streaming:
  streams:
    - name: "telemetry"
      sink:
        type: "kinesis"          # kinesis | kafka
        streamName: "device-telemetry"
        region: "us-east-1"
      buffer:
        # type defaults to "disk" (durable); use "memory" for a non-durable RAM buffer.
        path: "/greengrass/v2/work/{ComponentFullName}/stream-telemetry"
        segmentBytes: 1048576
        maxDiskBytes: 67108864
        onFull: "dropOldest"     # never silent -- always metered
      delivery:
        pollIntervalMs: 1000
Delivery semantics

Streaming is at-least-once: records carry (partitionKey, sequence) so downstream consumers can de-duplicate. Ordering is preserved per partition key. The overflow (onFull) default, dropOldest, is telemetry-typical and always emits a metric when it drops — it is never silent. Streaming is not a replacement for messaging, which remains your control plane.
